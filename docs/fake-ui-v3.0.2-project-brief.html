一、项目定位

fake-ui 不是“大而全”的传统面板，而是单 VPS 多出口代理编排系统，并在 v3.0.2 增加了可交付给客户使用的跨境本地服务发布能力。

产品视角

统一入口

一台线路稳定的 VPS 负责公网入口、证书、面板、订阅、Nginx HTTPS、Xray Reality 和 Hysteria2。客户或终端用户只需要访问统一域名或订阅地址。

多出口编排

同一台 VPS 可以维护多个 VLESS 节点，每个节点独立绑定直连、HTTP 上游或 SOCKS5 上游，节点名称可跟随真实出口 IP 和国家自动同步。

本地服务发布

客户的 Mac、Windows 或 Linux 机器没有公网 IP 也可以运行服务；后端 Agent 主动连接 VPS，VPS 负责把公网 HTTPS 或私有 TCP 流量转发到客户本地端口。

核心卖点：合作方只需要维护一台入口 VPS，就能同时提供普通代理节点、用户订阅、订单套餐、链上收款、公开网站发布、私有 SSH/TCP 通道和后端客户端配对交付。

二、项目结构

仓库按“面板入口、HTTP 路由、业务服务、配置编排、前端静态资源、脚本和测试”拆分。运行态数据、证书、真实域名、用户密码和订阅 token 不进入 Git。

三、VPS 总体网络拓扑

VPS 是整套系统的公网门面：浏览器、代理客户端、公开网站访问者和后端 Agent 都先连接 VPS，再由 Nginx、Xray、Hysteria2 或面板 API 分流。

TCP 80

由 Nginx 使用，负责 HTTP 访问、ACME 证书验证和必要的跳转。

TCP 443

公网 HTTPS 主入口，可承载面板、公开发布域名和 VLESS Reality 分流。

UDP 443

默认给 Hysteria2 使用；如果被占用，可在安装时调整 Hysteria2 端口。

127.0.0.1

面板、Nginx HTTPS upstream、Xray 内部端口都优先只在本机监听，降低暴露面。

四、本地服务发布链路

本地服务发布把“本地端口发布成公网域名”拆成 VPS 入口和后端 Agent 回连两段。后端机器不需要公网 IP，不需要路由器端口转发，只要能主动访问 VPS 即可。

服务发布能力
访问者浏览器 https://app.example.com
正常 HTTPS 网站访问
VPS Nginx 为 app.example.com 签发证书
反代到 127.0.0.1:portal
Xray Portal → Bridge portal inbound 接入请求
reverse outbound 回到客户机器
客户本地服务返回响应

公开 HTTPS 服务

需要域名

客户先把子域名 A/AAAA 解析到海外 VPS。面板创建发布服务时会校验域名是否解析到本 VPS、是否被面板域名、普通节点域名或其他已发布服务占用。保存后自动应用 Xray portal 和 Nginx HTTPS。

访问者
  -> https://public-domain
  -> VPS Nginx 127.0.0.1:portal_port
  -> VPS Xray portal
  -> 客户机器 Xray bridge
  -> 127.0.0.1:local_port

私有 TCP / SSH 服务

无需域名

SSH、数据库或私有 TCP 不需要公网域名，使用 VPS 的 portal 端口和跳板方式访问。SSH 默认作为全局私有 TCP 服务加入共享 Bridge，但不会在面板或本地控制台里显示具体域名，避免客户误解。

管理员或用户
  -> 先 SSH 到 VPS
  -> 再访问 127.0.0.1:portal_port
  -> Xray reverse
  -> 客户机器 127.0.0.1:22
对象 生成位置 唯一性规则 用途
UUID tunnel_catalog.normalize_tunnel 不能和普通代理用户 UUID、其他服务发布 UUID 重复 Bridge 连接 VPS Reality 入站时的身份。
Email tunnel:service-id 普通用户和服务发布 Bridge 分开命名 Xray clients 中区分普通代理流量和 reverse 流量。
Portal Tag tunnel-portal-service-id 每个服务唯一 VPS 上接收 Nginx 或 TCP 转发进入的入口。
Reverse Tag tunnel-reverse-service-id 每个服务唯一 把 portal 请求路由到对应后端 Bridge。
Portal Port 默认从 18081 起递增 每个服务唯一 Nginx 或私有 TCP 入口转发到 Xray portal。

为什么不会和普通代理节点搞混：虽然普通代理节点和本地服务发布 Bridge 可以共用 Reality SNI，例如 www.cloudflare.com，但 Xray 入站会按 UUID/client、email、reverse tag、portal inbound 和 routing rule 区分。普通代理用户不会带 reverse 配置，服务发布 Bridge 的 client 会带 reverse.tag。

五、普通代理节点链路

普通代理节点仍保持 fake-ui 原本能力：一台 VPS 创建多个 VLESS 节点，每个节点可以走本机直连、HTTP 上游或 SOCKS5 上游，并进入用户订阅。

代理能力
用户代理客户端 Shadowrocket / Clash / v2rayN 等
导入订阅链接
VPS VLESS Reality 统一入口、订阅、流量统计
SNI 可使用 www.cloudflare.com
节点出口 本机 direct
HTTP 上游
SOCKS5 上游

直连出口

VPS 自身 IP 作为出口，适合基础节点和低成本自用场景。

HTTP 上游

某个 VLESS 节点绑定 HTTP 代理上游，节点名称可根据上游出口国家同步。

SOCKS5 上游

某个 VLESS 节点绑定 SOCKS5 代理上游，适合多地区出口组合。

兼容边界：本地服务发布不会改变普通代理节点的订阅、UUID 或出口配置；服务保存、启用、停用、删除时只刷新 tunnel catalog、Xray portal 片段和 Nginx 发布配置，并在失败时回滚 catalog。

六、用户使用逻辑

v3.0.2 的目标是让客户“看一个页面、下一个包、跑一次安装脚本”，不用理解 Xray reverse、portal tag、Reality key 或 Nginx 配置细节。

客户视角

准备域名

公开网站发布需要客户把子域名解析到 VPS。私有 TCP/SSH 不需要域名，面板会明确标注。

管理员新增服务

填写公网域名、本地目标地址、目标端口、后端系统和 Bridge 模式。保存时自动校验并应用到 VPS。

下载后端客户端

推荐下载“后端客户端”通用配对包。一台后端机器对应一个 Agent，多个服务自动合并。

客户本机安装

客户在 macOS、Linux 或 Windows 上运行安装脚本。Agent bootstrap 后会拉取真实 xray-bridge.json。

本地控制台检查

打开 http://127.0.0.1:19090/ 查看 runtime、服务探测、日志、配置和本地代理兼容状态。

管理员页面应该怎么理解

  • 本地服务发布：一个公网域名或私有 TCP 入口，对应客户机器上的一个本地端口。
  • 后端客户端：一台客户机器运行的 Agent，可承载一个或多个服务发布。
  • 默认独立 Agent：每个服务一个 Bridge，适合隔离要求高的服务。
  • 高级共享 Agent：多个服务共用同一个 Bridge，适合一台机器上部署多个 Web 服务。
  • 导出总 JSON：把同一后端客户端下的全部服务合并导出，避免导入新服务时覆盖旧服务。

客户本地客户端应该怎么理解

  • 只监听本机：控制台默认只在 127.0.0.1:19090，不会暴露到公网。
  • 自动配对：配对包里的 token 只使用一次，成功后本地清空，面板只保存 hash。
  • 自动重启：安装脚本会写入 launchd、systemd 或 Windows 任务。
  • 代理兼容：如果本机开了 Shadowrocket、Clash、TUN，可点击“应用代理兼容”。
  • 手动导入：离线或排障时可导入总 JSON，客户端会合并服务列表。

七、后端 Agent 配对机制

配对机制让 release 包不包含任何客户域名、UUID、Reality 公钥或 portal 信息。真实配置只在客户下载面板生成的配对包并首次运行时获取。

交付安全

1. 面板生成配对包

管理员点击“下载后端客户端”，面板创建 agent_pairings 记录，只保存 token hash、过期时间、Bridge ID、bundle kind、platform 和 agent_id。

2. 客户运行安装脚本

脚本先执行 bootstrap-agent.py，读取 agent-profile.json，向面板 /api/agents/bootstrap 提交 token_id、pairing_token 和本机平台。

3. 面板返回真实配置

面板校验 token 未过期、未使用且 hash 匹配，然后生成 xray-bridge.json、bridge-dashboard.json 和 agent-state.json。

4. token 作废并启动服务

bootstrap 成功后面板标记 used_at，本地清空一次性 token，安装脚本继续校验 Xray 配置并注册后台服务。

为什么适合公开发布：GitHub release 中的通用客户端资产只包含示例 profile 和 bootstrap helper，不包含任何真实业务密钥；客户真实配置由自己的 fake-ui 面板现场生成。

八、域名、端口与模式规则

面板把“哪些域名能用、哪些服务需要域名、哪些服务不能显示域名”做成明确规则，减少客户误操作。

配置规则
场景 是否需要域名 面板行为 客户要做什么
公开 Web 服务 需要 只允许已解析到 VPS 且未被占用的域名保存；保存后自动签证书并写 Nginx。 把子域名 A/AAAA 解析到 VPS，再在面板选择或输入。
面板域名 系统保留 不会出现在服务发布可选域名里，避免覆盖面板自身。 不用于本地服务发布。
普通代理节点域名 系统保留 不会出现在服务发布可选域名里，避免影响订阅节点。 继续用于普通代理节点。
SSH / 私有 TCP 不需要 标注“TCP 无需域名”，SSH 默认全局只加入一次，不显示具体域名。 通过 VPS 跳板和 portal 端口访问。
共享 Bridge 按服务类型决定 多个 Web 服务默认复用已有共享后端客户端；导出时合并为总 JSON。 一台后端机器下载一个包即可。

九、部署与交付清单

从 VPS 到客户本机，交付可以分成“平台部署、域名接入、服务发布、Agent 安装、验证上线”五个阶段。

实施路径

平台方准备

  • 准备一台稳定 VPS，并开放 TCP 80、TCP 443、UDP 443。
  • 执行 scripts/install-fresh-vps.sh 安装 fake-ui。
  • 配置根域名、面板域名、Hysteria2 域名、VLESS 域名和证书邮箱。
  • 确认面板、普通代理节点、订阅、Xray、Hysteria2 正常。
  • 在“本地服务发布”页新增客户服务，保存后观察是否自动应用到 Xray / Nginx。

客户侧准备

  • 公开 Web 服务：把业务子域名解析到平台 VPS。
  • 本地机器运行真实服务，例如 127.0.0.1:8888 或 127.0.0.1:9999。
  • 下载面板生成的通用后端客户端配对包。
  • 按系统运行 macOS、Linux 或 Windows 安装脚本。
  • 打开本地控制台检查服务探测、Bridge 状态和公网访问。

macOS

安装脚本注册 launchd 服务，使用 ~/Library/LaunchAgents 和 ~/.fake-ui 目录保存运行文件。

Linux

安装脚本注册 systemd 用户服务或系统服务，适合服务器、NAS、迷你主机。

Windows

安装脚本使用 PowerShell 和计划任务，适合客户办公电脑或 Windows Server。

十、招商说明与商业价值

fake-ui 适合做成轻量自托管服务底座，也适合交付给小团队、开发者、工作室和需要把本地服务发布到海外 VPS 的客户。

合作价值

低成本入口

平台方只需要维护少量入口 VPS，就能复用证书、面板、订阅、用户、套餐、订单和本地服务发布能力。

高复购场景

客户新增一个网站、后台、Webhook、演示服务或私有 SSH，就能继续在同一面板里新增映射和续费。

交付门槛低

客户不需要理解反向代理、证书、NAT、Reality 或 Xray，只要解析域名、运行服务、下载客户端。

二开空间清晰

远程心跳、远程命令、日志回传、Agent 自动更新接口已预留命名空间，可以继续演进为商业 Agent 平台。

安全边界明确

本地控制台只监听 127.0.0.1；配对 token 一次性使用；GitHub release 不携带真实客户配置。

兼容原业务

本地服务发布不影响普通代理节点、用户订阅、Hysteria2、订单系统和已有 Nginx 原生部署模式。

推荐销售话术：“您只要把域名解析到我们的 VPS，再在自己的电脑或服务器上运行后端客户端，本地 8888、9999、22 等端口就可以变成公网可访问的 HTTPS 网站或私有远程通道。后续新增服务只在面板新增映射，同一台机器仍然只需要一个客户端。”

十一、验收与排障逻辑

排障要按链路分段，不要一上来重装。先看域名解析，再看 VPS Nginx/Xray，再看后端 Agent，再看本地服务。

运维手册
现象 优先检查 判断方式 常见处理
公网访问 502 VPS Nginx 到 portal 是否通、Bridge 是否在线 面板点“应用到 VPS”；本地控制台看 runtime 和日志 重启后端客户端，确认总 JSON 包含所有服务。
新增服务后旧服务消失 是否导入了单服务 JSON 本地控制台服务列表只剩一个 使用“导出总 JSON”或配对安装包，确保同一 Agent 下服务合并。
域名不能选择 DNS 是否解析到 VPS、是否被保留或占用 面板域名资格提示 修正 A/AAAA 解析，避开面板域名和普通节点域名。
本机开代理后不通 Bridge 回连是否走了 TUN/fake-ip 路由 本地控制台“本地代理兼容”状态 点击“应用代理兼容”，再重启 Bridge。
SSH 看不见域名 是否为私有 TCP 服务 面板标注 TCP 无需域名 这是预期行为，通过 VPS 跳板和 portal 端口访问。
